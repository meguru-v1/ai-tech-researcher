# precheck — コミット前のマルチエージェント・レビュー

単一AIに「大丈夫?」と聞くのは自己確証バイアスで危険。**別々のレンズを持つサブエージェントで並列レビューし、各指摘を再現/実機で検証してから**確定だけ報告する。`/vibe-check` が「観点（What）」、これが「実行（How）」。

## いつ走らせるか
コミットを提案する直前で、差分が次のどれかに触れるとき必須：認証・認可・公開範囲／ユーザーデータ・PII／秘密情報・env／DBスキーマ・マイグレーション・破壊的スクリプト／外部fetch・LLM/DB課金／公開UIのデータ取得。トリビアルな文言/CSSのみなら省略可。

## 手順
1. **対象の確定**: `git diff`（未ステージ＋ステージ）と新規ファイルを把握。変更ファイルと「何に触れているか」を1行で言語化。
2. **並列レビュー**: Agent ツールで**独立した**サブエージェントを同時に投入（各々に差分と着眼を渡す）。最低この3レンズ：
   - **security-auditor**: IDOR/認可漏れ（クライアント供給idの信用）・秘密の直書き/ログ露出・SQLi/XSS/SSRF・**ブラックリスト方式の見落とし**・エラー詳細の露出。出典の観点は [[vibe-check]] 第一条。
   - **migration-checker**: schema 変更と利用側の整合（未コミットschemaで本番だけ壊れる罠）・冪等性/トランザクション境界・破壊的UPDATE/DELETEの影響範囲・本番DBへの誤接続。[[vibe-check]] 第四条。
   - **bug-finder**: ロジック誤り・境界値・null/undefined・例外時の無限ローディングや状態不整合・N+1。[[vibe-check]] 第五条。
   - （任意）**legal/cost**: スクレイピング/本文公開・ライセンス・LLM/DB課金のループ上限。[[vibe-check]] 第二/三条。
3. **検証（鵜呑み禁止）**: 各指摘を**コードで再現・確認**する。「容疑」を「判決」にするのは実機/再現。確証が持てない指摘は破棄するか「未確認」と明記。可能なら該当パスを実際に叩く（読み取り専用）。
4. **報告**: 確定した問題だけを `file:line` ＋「なぜ問題か」＋「直し方」で列挙。重大度（🔴/🟡）付き。問題なしならその旨。
5. **ゲート**: 🔴 が残るうちはコミットを提案しない。直す or ユーザーが了承するまで止める（[[vibe-check]] 第六条＝まず止める）。

## 注意
- サブエージェントの指摘は**多角化のため**であって正解ではない。多数決でなく**再現で**採否を決める。
- 破壊的migration/スクリプトは、本番に流す前に読み取り専用クエリ（PRAGMA/SELECT）で前提を確認してから。
- 過去の実例: `corpus_health` が除外ブラックリストから漏れてID総当たりで露出（→ホワイトリスト化）。この手の「列挙で漏れる」系を security-auditor が拾えるよう差分の文脈を渡す。
